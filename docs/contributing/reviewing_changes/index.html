    Reviewing Changes #

    The review process helps to ensure high quality and catch problems in contributions. When acting as a reviewer, you should see your position as sharing your knowledge and working with the contributors to achieve the best possible result. Reviewing should not be an adversarial process.

    As a reviewer you should

    1. Use the GitHub review system to check the diffs of all source files.
    2. Clone the branch, build the handbook locally (using hugo server --minify) and inspect the changes using your browser. Not all bugs will be caught by CI and not all changes will be obvious in the source files.
    3. Use line comments where you have comments or questions about particular changes so that the context is clear.
    4. Use line comments to suggest changes (using the Â± button).
    5. Submit your review making sure to choose “comment”, “approve” or “request changes” as appropriate.
    6. Work together with the contributors to resolve any conversations. All conversations must be resolved before merging.
    7. Update your review status to “approve” when you are happy.
    8. When all reviewers are happy, merge the pull request.
